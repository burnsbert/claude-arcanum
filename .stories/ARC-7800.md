# ARC-7800: Implement Maestro Semi-Autonomous Development Pipeline

## Summary

Build Maestro, a 10-phase semi-autonomous development pipeline that takes a story (from Jira or a local file) and drives it from research through implementation, code review, and PR creation. Maestro coordinates specialized agents — scout, planner, developers, validator, reviewers — to ship a complete story with minimal human intervention.

Maestro must be tech-stack agnostic: no assumptions about language, framework, test runner, or cloud provider. Every agent relies on the scout's codebase research to determine project conventions.

## Architecture

### Naming Convention

- `arc-maestro` / `arc-maestro-review` — User-facing commands (`arc-` prefix)
- `ca-maestro-*` — Internal utility agents (`ca-` prefix)

### File Organization

All Maestro scratch files live in a `.maestro/` directory at the project root (must be added to `.gitignore`). Three files per story, plus optional demo artifacts:

```
.maestro/
  context-{STORY-ID}.md    # Status dashboard — where things stand now
  todo-{STORY-ID}.md       # Task list with difficulty/type tags
  diary-{STORY-ID}.md      # Narrative log — how we got here
  demo/                    # Demo artifacts (if frontend tasks produce them)
```

#### Context File — Status Dashboard

**`.maestro/context-{STORY-ID}.md`** — Single source of truth for project status. Reading this file tells you everything important about where the project stands right now.

```markdown
# Maestro Context: {STORY-ID}

## Story Details
**Source**: {Jira: STORY-ID | File: filepath}
**Title**: {story title}
**Type**: {Story/Bug/Task}
**Priority**: {priority if available, or "Not specified"}
**Status**: {status if available, or "Not specified"}

**Description**:
{full story description}

**Acceptance Criteria**:
{acceptance criteria if available}

## Current Status
**Phase**: Phase 1: Initialize
**Progress**: Story initialized
**Started**: {timestamp}
**Last Updated**: {timestamp}
**Next Action**: Launch scout agent for research

_Agents update Phase, Progress, Last Updated, and Next Action as work progresses_

## Research Findings
_Populated by scout agent_

## Task Progress

### Completed Tasks
_Track completed tasks with summaries as development progresses_

### Current Task
_Track the task currently being worked on_

### Pending Tasks
_Remaining tasks from the plan_

## Agent Outputs
_Agent outputs recorded here_

## Blockers
_None at initialization_

## Decisions
_Key decisions recorded here_
```

#### Diary File — Narrative Log

**`.maestro/diary-{STORY-ID}.md`** — The story of how the project unfolded. Append-only, chronological. All agents read the diary before starting work and write to it when relevant. See Context Management acceptance criteria for methodology.

**Cross-story lookup:** Diary files from past stories are NOT read by default. Agents only consult past diaries when stuck on a task or searching for an answer that might exist in prior project history.

#### Todo File — Task List

**`.maestro/todo-{STORY-ID}.md`** — Task list created by the planner.

### The 10 Phases

| Phase | Name | What Happens | User Checkpoint? |
|-------|------|-------------|------------------|
| 1 | Initialize | Fetch story, create branch, create context files | No |
| 2 | Scout | Research codebase patterns, conventions, test coverage | No |
| 3 | Questions | Present scout's ambiguities to user for answers | Yes (if ambiguities exist) |
| 4 | Plan | Break story into tasks with difficulty/type/TDD structure | No |
| 5 | Review | Quality gate: plan-reviewer vets and improves the plan | No |
| 6 | Approve | User reviews and approves the plan | Yes |
| 7 | Develop | Implement tasks one-by-one with mandatory validation | Yes (only on blocker) |
| 8 | Code Review | Two-pass review: generate concerns then batch-vet | No |
| 9 | Respond | Fix bugs with regression tests, address other concerns | No |
| 10 | Complete | Commit review fixes, push, create PR | No |

Phases 1-7: `/arc-maestro`. Phases 8-10: `/arc-maestro-review`.

### Agent Roster

| Agent | Model | Role |
|-------|-------|------|
| `ca-maestro-scout` | Opus | Codebase researcher |
| `ca-maestro-planner` | Opus | Task decomposer |
| `ca-maestro-plan-reviewer` | Opus | Quality gate — vets and improves plan before user sees it |
| `ca-maestro-dev-doer` | Sonnet | Standard implementer — difficulty 1-6 tasks |
| `ca-maestro-senior-dev-doer` | Opus | Complex task specialist — difficulty 7+ and escalations |
| `ca-maestro-frontend-dev-doer` | Opus | UI/UX specialist — `[Type: frontend]` tasks at any difficulty |
| `ca-maestro-devops-dev-doer` | Opus | Infrastructure specialist — `[Type: devops]` tasks at any difficulty |
| `ca-maestro-task-validator` | Sonnet | Strict pass/fail gate — COMPLETE or INCOMPLETE |
| `ca-maestro-code-review` | Opus | Two-pass reviewer with enhanced bug-finding |
| `ca-maestro-code-review-responder` | Opus | Addresses vetted concerns, generates completion report |

`ca-code-review-validator` (already exists in claude-arcanum) — Batch validates review feedback items. Used by code-review agent in Pass 2.

## Design Decisions

### Tech-Stack Agnostic (CRITICAL)

- **No framework assumptions.** Don't assume React, Vue, Angular, Django, Rails, Spring, Laravel, or any other framework.
- **No language assumptions.** Don't hardcode test commands or file patterns for any language. No `npm test`, no `vendor/bin/phpunit`, no `pytest` in agent prompts.
- **No cloud provider assumptions.** Don't prescribe AWS, GCP, or Azure services.
- **No package manager assumptions.** Don't hardcode npm, pip, composer, cargo, etc.
- **Scout is source of truth.** Every agent relies on scout research to know what tools, patterns, conventions, and test frameworks the project uses.

### Agent Routing (Phase 7)

Two-dimensional routing:
1. **Type tag** (checked first): `[Type: frontend]` → frontend specialist, `[Type: devops]` → devops specialist — regardless of difficulty
2. **Difficulty rating** (fallback for untagged tasks): 7+ → senior-dev-doer, 1-6 → dev-doer

Failure handling:
- Specialist tasks retry with the same specialist (no cross-agent escalation)
- Untagged tasks escalate from dev-doer to senior-dev-doer after 2 failures
- Development halts after 3 failures on any single task — user decides next step

### TDD by Convention

TDD is mandatory for file types with established test patterns (as discovered by the scout). TDD is NOT forced on file types that aren't typically tested in the project. The scout documents which file types have test patterns; the planner marks tasks accordingly; developers follow.

### Mandatory Task Validation

Every task requires validation before proceeding. No exceptions. The validator:
- Runs tests independently (doesn't trust dev-doer claims)
- Checks full scope completion (no scope reduction)
- Returns binary COMPLETE or INCOMPLETE

### Input Sources

Two input types:
- **File-based stories**: Contains `/` or starts with `./`/`~` or ends with `.md`/`.txt`. Story ID becomes `FILE-{FILENAME-UPPERCASE}`. If a filename is provided, assume file — skip Jira entirely.
- **Jira tickets**: Pattern `[A-Z]+-[0-9]+`. Fetched via `mcp__jira__get-issue` or `mcp__jira-read__get-issue`. If Jira MCP tools are not available or the ticket is not found, look for a matching file in `.stories/` or the project root before failing.

### Resume Capability

`/arc-maestro` resumes from where it left off by reading the "Current Status" section of the context file. `/arc-maestro-review` always starts fresh from Phase 8.

### Visual Verification During Development

For stories with UI components or where tests alone wouldn't catch visual/interaction problems, Maestro should use browser automation (Playwright MCP or Claude Code browser integration) during Phase 7 to verify that implemented UI actually looks and interacts correctly. This is not just screenshot capture for a PR — it's active visual QA during development that can identify problems and create new fix tasks.

Key constraints:
- **Local only.** Everything runs against the local dev environment. Never deploy to external systems unless the user explicitly asks.
- **Not every story.** The scout and planner determine whether visual verification is appropriate based on story type and UI involvement.
- **Creates tasks when needed.** If visual verification reveals problems (broken layout, missing states, interaction issues), new tasks should be created to address them.

### Continuous Execution

Maestro runs autonomously between user checkpoints. No "proceed?" pauses between Phases 2→4→5 or between individual tasks in Phase 7. The user can interrupt if needed.

## Implementation Tasks

### 1. Create the `arc-maestro` command

Create `commands/arc-maestro.md` — orchestrator for Phases 1-7.

**Phase 1: Initialize**

Input parsing:
- Check if argument contains `/`, starts with `./`/`~`, or ends with `.md`/`.txt` → file path
- Check if argument matches `[A-Z]+-[0-9]+` → Jira ID
- If neither, prompt user for input

Branch handling:
- Get current branch via `git branch --show-current`
- If current branch starts with lowercase story ID → already on correct branch
- If matching branch exists → ask user whether to switch
- If no matching branch → create from base branch (`development` > `develop` > `main` > `master`, check in order), name it `{lowercase-story-id}-{2-4-word-kebab-description}`

Resume check:
- If `.maestro/context-{STORY-ID}.md` exists, read "Current Status" section
- Show current phase/progress, ask: resume or start fresh?
- Resume jumps to appropriate phase; fresh start deletes existing files

Story loading:
- Jira: fetch via MCP tools, extract title, description, ACs, type, status, priority
- File: read contents, extract title (first heading), description (body), ACs (if labeled section exists), default type to "Story"

Create context file from template above.

**Phase 2: Scout**

Launch `ca-maestro-scout` via Task tool. Prompt: research codebase for story {STORY-ID}, update Research Findings in context file.

Review scout's findings. Check for unanswered questions or story gaps.

**Phase 3: Questions (conditional)**

If scout has unanswered questions or story gaps → present to user, wait for answers, record in Decisions section.

If none → skip to Phase 4.

**Phase 4: Plan**

Launch `ca-maestro-planner` via Task tool. Creates `.maestro/todo-{STORY-ID}.md`.

**Phase 5: Review**

Launch `ca-maestro-plan-reviewer` via Task tool. Reviewer applies improvements directly to todo file before user sees it.

**Phase 6: Approve**

Present the improved plan to user. Handle:
- Approved → proceed to Phase 7
- Changes requested → apply changes, re-present
- Loop until approved

**Phase 7: Develop**

For each task in the todo list:

A. **Route task**: Check type tag first, then difficulty rating (routing rules above)

B. **Launch dev agent**: Via Task tool with appropriate `subagent_type`. Include: task description, story ID, task number/total.

C. **Mandatory validation**: Launch `ca-maestro-task-validator` via Task tool. NEVER skip. NEVER proceed without validator response.

D. **Handle result**:
- COMPLETE → mark task done in todo file, update Task Progress in context file, move to next task
- INCOMPLETE + failures < 3 → retry (escalate to senior if untagged and failure count = 2)
- INCOMPLETE + failures = 3 → HALT, document blocker, ask user: implement manually / adjust scope / skip

E. **New task discovery**: During development, agents may discover new requirements — technical dependencies the scout missed, edge cases that only become apparent during implementation, visual issues caught by browser verification, etc. When this happens, new tasks can be added to the todo list and executed as part of the current Phase 7 run. Log the discovery and rationale in the diary.

After all tasks: stage changes, commit (`{STORY-ID}: {title}`), push, report completion, point user to `/arc-maestro-review`.

**Commit message format:**
```
{STORY-ID}: {Story title}

Implemented all tasks:
- {Brief summary}

All tests passing. Ready for code review.

Generated with Maestro
Co-Authored-By: Claude <noreply@anthropic.com>
```

### 2. Create the `arc-maestro-review` command

Create `commands/arc-maestro-review.md` — handles Phases 8-10.

**Story ID detection**: Accept as argument, or find most recent context file via `ls -t .maestro/context-*.md 2>/dev/null | head -1`.

**Phase validation**: Read context file. Development must be complete (Phase 7 finished, not just in progress). If Phase 7 is not complete, stop and tell user to finish development first.

**Always fresh**: This command always starts from Phase 8, even if previously run.

**Phase 8**: Launch `ca-maestro-code-review` via Task tool. Read the review report.

**Phase 9**: If concerns found → launch `ca-maestro-code-review-responder` via Task tool. If no concerns → skip to Phase 10.

**Phase 10**:
- Commit any review fixes (`{STORY-ID}: Address code review feedback`)
- Push
- Create PR via `gh pr create` with summary, changes, testing results, story reference
- Display final report with PR URL

**PR body format:**
```markdown
## Summary
{Brief description of what was implemented}

## Changes
{List of main changes}

## Testing
- All unit tests passing
- Code review: {X} bugs fixed with regression tests, {Y} concerns addressed

## Story
{STORY-ID}

Generated with Maestro
```

### 3. Create the `ca-maestro-scout` agent

Create `agents/ca-maestro-scout.md` — Opus, tools: Glob, Grep, Read, Bash, WebFetch, WebSearch, Edit, TodoWrite.

**Process:**

1. **Analyze story**: Determine type (FE-only, BE-only, full-stack, bug fix) based on labels, scope, ACs. Run story refinement: check AC completeness, identify edge cases, flag gaps and ambiguities.

2. **Generate 5-10 implementation questions**: What patterns exist? What components/services are affected? What edge cases need handling? What testing approach? What could break?

3. **Research priority areas** (in order):
   - `guides/` directory — conceptual understanding of codebase aspects. Document whether relevant guides were found.
   - `bugfinder.md` — common pitfalls, areas prone to errors
   - `.guide.md` / `CLAUDE.md` — project-specific knowledge
   - **Test coverage as documentation** (CRITICAL):
     - Identify which FILE TYPES have established test patterns → TDD mandatory for these
     - Identify which file types do NOT have test patterns → don't force tests
     - Extract: business rules, edge cases, mock patterns, coverage gaps from existing tests
   - Codebase patterns — similar features, relevant services/models/controllers

4. **Answer questions and investigate follow-ups**

5. **Generate research report** with these sections:
   - Story Analysis (type, dependencies, scope, refinement findings with gap list)
   - Questions Investigated (answer, citation `file.ext:function:line`, confidence High/Medium/Low)
   - Key Findings (patterns found, similar features, constraints)
   - Test Coverage Insights:
     - File types WITH established test patterns (TDD MANDATORY) — list each with pattern description
     - File types WITHOUT test patterns (don't force tests) — list each
     - Business rules found in tests
     - Coverage gaps
   - Unanswered Questions (with "needs: code review / PM clarification / architecture decision")
   - Ambiguities in Story (with impact and clarifying question)
   - Recommended Approach
   - Citations Summary

6. **Update context file**: Replace Research Findings placeholder with full report. Update Current Status.

7. **Update diary**: Log key research discoveries, surprises, and any gaps that could affect later work.

**Key principles**: Work autonomously (no asking user during research). Be thorough — this research drives the entire implementation. Provide citations. Flag ambiguities.

### 4. Create the `ca-maestro-planner` agent

Create `agents/ca-maestro-planner.md` — Opus, tools: Read, Write, Edit, Glob, Grep, TodoWrite.

**Process:**

1. **Read context file completely**: story details, scout research (including guides/ findings), user decisions

2. **Analyze work by story type**:
   - Full-stack: data layer + business logic + API + frontend + integration
   - Backend-only: data layer + business logic + API + tests
   - Frontend-only: components + state + UI integration + tests
   - Bug fix: investigation + test to verify bug + fix + regression test

3. **Structure tasks** in logical order, organized by layer/concern. Each task must include:
   - Clear description of what to implement
   - `[Difficulty: N/10]` rating
   - `[Type: frontend]` or `[Type: devops]` tag when applicable
   - Implementation notes with scout citations

**Difficulty scale:**

| Rating | Description | Examples |
|--------|-------------|---------|
| 1-2 | Trivial | Config changes, simple string updates |
| 3-4 | Easy | Standard CRUD, following clear patterns |
| 5-6 | Moderate | Some complexity, multiple components |
| 7 | Challenging | Non-trivial logic, careful edge cases |
| 8 | Complex | Architectural decisions, multi-system integration |
| 9 | Very Complex | Intricate business logic, performance-critical |
| 10 | Expert | Novel problems, no clear patterns to follow |

**Specialist type tagging rules:**

- `[Type: frontend]` — tasks primarily involving UI components, styling, state management, accessibility, browser APIs, frontend testing. Use for ANY frontend work regardless of difficulty.
- `[Type: devops]` — tasks primarily involving infrastructure, CI/CD, containerization, monitoring, security config, networking, build scripts. Use for ANY infra work regardless of difficulty.
- No tag — backend business logic, API endpoints, database work, general-purpose tasks. Routed by difficulty.

**Story-level type inference**: If scout identifies story as "FE-only", most tasks should be `[Type: frontend]`. If "DevOps", most should be `[Type: devops]`. If "full-stack", tag each task individually.

**TDD rules from scout research:**
- For file types WITH test patterns: mark task notes as "TDD MANDATORY"
- For file types WITHOUT test patterns: don't force tests
- Prefer combined "Test & implement" tasks over separate test/implement pairs

**Non-automatable work**: If work requires human action (manual browser testing, visual verification, external service config), create "Remind user to..." tasks at END of list, marked `[Difficulty: 1/10]`. Use sparingly.

**Todo file format:**
```markdown
# Implementation Plan: {STORY-ID} - {Story Title}

**Story Type**: {type from scout}
**Estimated Tasks**: {count}
**Approach**: {detailed summary}

## Key Decisions
- {Decision with rationale}

## Task Breakdown

### {Section Name}
- [ ] {Task description} [Type: frontend] [Difficulty: N/10]
  Notes: {TDD requirements, citations, approach guidance}

## Implementation Notes
**Patterns to Follow**: {with citations}
**Avoid**: {anti-patterns}
**Test Coverage Goals**: {based on repo conventions}

## Success Criteria
- [ ] All acceptance criteria met
- [ ] Test coverage matches repo standards
- [ ] No code duplication introduced
- [ ] Follows established patterns
```

**Note**: Code review is NOT part of the plan — handled separately by `/arc-maestro-review`.

4. **Update context file** with planning status.

5. **Update diary**: Log planning approach, key decisions, and any trade-offs considered.

### 5. Create the `ca-maestro-plan-reviewer` agent

Create `agents/ca-maestro-plan-reviewer.md` — Opus, tools: Read, Edit, Grep, Glob, TodoWrite.

**Review dimensions:**

1. **Scout research quality**: Completeness (all ACs researched?), accuracy (spot-check citations with Read tool), gaps (missed questions? missed edge cases? checked guides/?)

2. **User answer quality**: Clear and actionable? Consistent with each other and with ACs? Any follow-ups needed?

3. **Plan quality**:
   - Coverage: every AC has tasks? every scout-identified edge case has test tasks? error scenarios covered? permissions covered?
   - TDD adherence: file types WITH test patterns have "TDD MANDATORY" in notes? file types WITHOUT patterns aren't forced to have tests?
   - Pattern following: leverages scout citations? avoids duplication?
   - Task quality: specific and actionable? properly sized (1-3 hours)? logical order? dependencies clear? enough detail for user to review?
   - Risks: missing tasks? tasks to combine? tasks to split?

4. **Cross-cutting concerns**: security (auth, input validation, injection prevention), performance (large datasets, N+1 queries, caching), error handling (tested? user-friendly messages? logging?), accessibility (if UI — keyboard nav, screen readers, ARIA)

5. **Automated testing only**: All testing must be automatable. Remove any manual testing tasks. Convert to automated equivalents.

**CRITICAL: Apply ALL improvements directly to `.maestro/todo-{STORY-ID}.md`:**
- Add missing tasks
- Remove unnecessary tasks (including any code review or manual testing tasks)
- Improve descriptions with more detail
- Add/fix TDD requirements based on file type patterns
- Add citations
- Split oversized tasks, combine overly granular tasks
- Add security/performance/error handling tasks where missing

Add review summary to context file under "Plan Review" section.

### 6. Create the `ca-maestro-dev-doer` agent

Create `agents/ca-maestro-dev-doer.md` — Sonnet, tools: Read, Write, Edit, Bash, Grep, Glob, MultiEdit, TodoWrite.

**Process:**

1. **Read task, context, and diary**: task description, scout research, patterns/citations, user decisions, related tasks, and diary entries from earlier work on this story

2. **Check guides/ note**: Scout has already determined if guides/ documentation is relevant. Use scout's findings in context file — don't re-read guides/.

3. **Implement following TDD** (when task notes say TDD MANDATORY):
   - "Test & implement" tasks: write test FIRST → run (should FAIL) → implement minimum to pass → run (should PASS) → refactor
   - Separate TEST tasks: write test → run (should FAIL) → verify fails for right reason
   - Separate IMPLEMENTATION tasks: find corresponding test → review expectations → implement → run (should PASS)

4. **Handle testing by file type**:
   - If file type has established test pattern (from scout) → TDD mandatory, write test first
   - If file type does NOT have test pattern → don't force tests, focus on implementation

5. **Run tests and verify**: Identify relevant tests (use Glob/Grep to find test files for changed code). Run them. ALL must pass (or fail correctly for test-only tasks). No skipped tests. Include output in summary.

6. **If tests fail**: Read failure carefully → fix implementation (not the test) → re-run → repeat until pass. If stuck, document what was tried and mark incomplete.

7. **Document with implementation summary:**
```markdown
## Task Implementation Summary

**Task**: {description}

**What was implemented**:
- {change 1}
- {change 2}

**Files modified/created**:
- `path/to/file` - {what changed}

**Tests run**: {command}

**Test results**:
- All tests passed: {yes/no}
- Total tests: {number}
- Any skipped: {yes/no}

**Patterns followed**: Used {pattern} from `citation:line`

**Notes**: {decisions, blockers}
```

**Key rules**: Implement ONE task. Use scout citations. Be honest about completion. Don't guess if unclear — report "Task unclear" and let validator mark incomplete. Write to the diary when you discover something that could affect later tasks.

### 8. Create the `ca-maestro-frontend-dev-doer` agent

Everything from dev-doer (including diary read/write), plus frontend domain expertise:

### 10. Create the `ca-maestro-task-validator` agent

Create `agents/ca-maestro-task-validator.md` — Sonnet, tools: Read, Bash, Grep, Glob.

**CRITICAL FIRST STEP: Read task history from context file's "Task Progress" section.**
- Check "Completed Tasks" — see what PREVIOUS tasks already accomplished
- Check "Current Task" — understand what THIS specific task should do
- This prevents false positives (e.g., blaming current task for files created by a previous task)

**Validation process:**

1. **Understand requirements**: Read full task scope from todo file. What does "done" look like?

2. **Review claims**: Read dev-doer's summary. What files changed? What test results?

3. **Verify implementation** (with task history awareness):
   - Read changed files, verify changes exist
   - Cross-reference with Completed Tasks — don't attribute previous work to current task
   - Check for: full scope implemented? TODOs left? commented-out code? scope reduction? hardcoded values that should be dynamic?

4. **Run tests independently** (CRITICAL — don't trust dev-doer output):
   - **Efficiency rule**: Only run tests that directly relate to changed code. Do NOT run the complete test suite.
   - Find test files for changed code using Glob/Grep
   - Run targeted tests
   - Capture complete output

5. **Verify test results**:
   - ALL pass? (not "most" — ALL)
   - Any skipped? (skipped = INCOMPLETE)
   - Tests actually cover new functionality? (not just placeholders)
   - Assertions check the right things?

6. **TDD exception**: NEW failing tests are ALLOWED if ALL conditions met:
   - Task explicitly says "write test" or "add test" (not "implement feature")
   - Clear NEXT task in todo list to implement the functionality
   - Test is well-written and tests the right thing
   - Functionality does NOT already exist

7. **Verdict**: COMPLETE or INCOMPLETE. Nothing in between.

**COMPLETE requires ALL of:**
- Full task scope implemented
- All tests pass (zero failures, zero skipped) — or TDD exception applies
- Tests are meaningful and cover functionality
- No scope reduction, no shortcuts
- Patterns from scout research followed

**INCOMPLETE if ANY of:**
- Partial implementation
- Tests failing or skipped (without TDD exception)
- Missing required tests
- Scope reduced
- TODOs, commented code, hardcoded values
- Patterns not followed

**Output format for COMPLETE:**
```markdown
STATUS: COMPLETE

## Validation Summary
{What was verified}

## Test Results
```

### 11. Create the `ca-maestro-code-review` agent

Create `agents/ca-maestro-code-review.md` — Opus, tools: Glob, Grep, Read, Bash, Task, TodoWrite.

**Two-pass methodology:**

**PASS 1: Generate comprehensive concerns**

1. Read context file and todo list. Check for `bugfinder.md` (project-specific bug patterns). Get git diff (`git diff main...HEAD`).

2. Review ALL dimensions:
   - **Testing**: unit test coverage, API test coverage, changed interfaces → updated tests?, test quality, edge cases, conventions
   - **Data layer**: migrations needed/correct/safe? rollback plan? data integrity?
   - **Performance**: response time impact? N+1 queries? caching? resource usage?
   - **Bugs** (enhanced rigor — see below)
   - **Security**: auth checks, access control, input validation, injection prevention
   - **Frontend** (if applicable): scoped styles, responsive, accessibility
   - **Code quality**: project conventions, DRY, error handling, logging, readability
   - **Integration**: breaking changes, API contracts, dependencies

3. **Bug-finding rigor:**
   - **Executable failure path required** for every bug: "If input X and state Y, then code Z produces wrong result W"
   - **Tri-state logic detection**: For nullable boolean params (especially filters like `is*`, `include*`, `exclude*`):
     1. Trace through code with each value: true, false, null/undefined
     2. For APIs: what items returned in each case? what are counts?
     3. Does count match returned items for ALL three cases?
   - **Framework contract verification**: Before flagging `needsX`/`missingX`/`lacksX` methods, MUST read the framework/base class implementation. Understand what true/false means in that context.
   - **Boolean expression verification**: Quote the expression, substitute actual values, evaluate step-by-step. Show arithmetic if affects counts/filters. Mismatch = BUG.

4. **Line number verification**: Before citing ANY line reference, use Read tool to view the actual file at that line. Quote the actual code to prove verification.

5. **Concern format** — every concern MUST include:
   - File path and verified line number
   - Type: Bug / Security / Test Gap / Performance / Quality / Documentation
   - Confidence: High / Medium / Low
   - For bugs: executable failure path
   - Suggested fix

6. **Focus on real issues**: Would you stop a PR merge for this? If no, don't flag it. No nitpicks, style preferences, or micro-optimizations.

**PASS 2: Batch validate all concerns**

Compile ALL Pass 1 concerns into a single list. Launch `ca-code-review-validator` via Task tool with the complete list (single call, not per-item). Process validator results:
- KEEP: concerns endorsed by validator
- REMOVE: false positives, nitpicks, already-fixed items
- CLARIFY: concerns needing more context

**Output: Vetted review report** stored in context file:
```markdown
# Code Review Report: {STORY-ID}

## Summary
Files Changed: {count}
Total Concerns (Pass 1): {count}
Vetted Concerns (Pass 2): {count}
  - Bugs: {count}
  - Critical: {count}
  - Important: {count}
  - Minor: {count}

## Positive Aspects
- {Good patterns followed} - `file:line`

## Bugs (Must Fix)
{Each with failure path, evidence, fix, regression test note}

## Critical Issues (Must Fix)
{Security, data integrity, breaking changes}

## Important Concerns (Should Fix)
{Test gaps, performance, quality}

## Minor Suggestions (Consider)
{Documentation, style}

## Validation Summary
Endorsements: {count}, Removed: {count}, Adjustments: {summary}

## Next Steps
{Ready to Complete / Requires Fixes}
```

### 12. Create the `ca-maestro-code-review-responder` agent

Create `agents/ca-maestro-code-review-responder.md` — Opus, tools: Read, Write, Edit, Bash, Grep, Glob, MultiEdit, TodoWrite.

**Process:**

1. **Handle bugs FIRST** (special priority):
   For each bug:
   a. Understand the executable failure path
   b. Implement the fix (minimal, focused)
   c. Add regression test if code has test coverage (test reproduces failure scenario, would fail without fix, passes with fix). If no existing test coverage for this code, note "no regression test — code has no existing test coverage"
   d. Run all related tests — must pass
   e. Document: bug description, failure path, fix applied, regression test, test results

2. **Process non-bug concerns** using decision tree:
   - Critical issue (security, data, breaking) → FIX IMMEDIATELY
   - Objectively wrong code / missing required tests → FIX
   - Materially improves quality (readability, prevents future bugs, noticeable perf) → FIX
   - Quick fix (< 5 minutes) → FIX
   - In scope but time-consuming → DOCUMENT as follow-up
   - Pre-existing / out of scope → DOCUMENT
   - Style preference → DISMISS with explanation

3. **For each concern, document the decision:**
   ```markdown
   ## Response to Concern #{N}
   **Concern**: {description}
   **Decision**: {FIX / DOCUMENT / DISMISS}
   **Action Taken**: {what was done and why}
   **Files Modified**: {if applicable}
   ```

4. **Run final verification**: All tests must pass. No skipped tests. Linting passes if configured.

5. **Generate completion report:**
   ```markdown
   # Maestro Completion Report: {STORY-ID}

   ## Story Complete

   **Ticket**: {STORY-ID}
   **Title**: {title}
   **Started**: {timestamp}
   **Completed**: {timestamp}

   ## Summary

   ### Tasks Completed: {count}, all validated
   ### Code Changes: {files modified/added/deleted, lines changed}

   ### Quality Gates Passed
   - Scout research completed
   - Plan reviewed and approved
   - All tasks implemented and validated
   - Code review performed
   - Concerns addressed
   - Final tests passing

   ## Code Review Response
   - Fixed: {count} ({list briefly})
   - Documented: {count} ({list briefly})
   - Dismissed: {count} ({list briefly})

   ## Final Test Results
   {test output}
   Tests: {count}, Passed: {count}, Failed: 0, Skipped: 0

   ## What Was Built
   {2-3 sentence summary}
   {Key files changed with descriptions}

   Generated with Maestro
   ```

**Diary**: Log bugs found, fixes applied, and any systemic patterns observed during review.

**Halt conditions:**
- \>10 critical/important concerns → suggest manual review, don't try to fix everything autonomously
- Concern is valid but unfixable → document blocker, don't mark complete
- Tests fail after fixes (2 attempts) → halt, show output, ask for help
