# PR Review File Format

The orchestrator writes review results to `.code-reviews/` in this exact format. The file is both human-readable markdown and machine-parseable by `review-to-html.py`.

## Filename Convention

`.code-reviews/<pr-number>-<YYYY-MM-DD-HHMM>.md`

Examples:
- `.code-reviews/7654-2026-04-07-1430.md`
- `.code-reviews/branch-2026-04-07-1430.md` (for branch-diff mode)

## Format

```markdown
---
title: "PR title or branch name"
risk: 6
recommendation: request-changes
pr_url: https://github.com/owner/repo/pull/123
head_ref: feature/my-branch
---

## Intent
[2-3 sentence problem-first synthesis]

## Changes
[High-level summary of what changed]

## Risk
[One sentence explaining the key risk drivers]

## Results
[What the review found - overall quality signal]

## Positives
- [Positive point] `file.ext:line`
- [Positive point] `file.ext:line`

## Critical Issues

### C1. [Title]
**File**: `path/to/file.ext:line`

[Problem description paragraph]

**Suggested change**: [Solution text]

### C2. [Title — bugfinder-confirmed example]
**File**: `path/to/file.ext:line`
**Bugfinder**: Confirmed (BF-1)

[Problem description — this finding was flagged by both pass1 and the local bugfinder independently]

**Suggested change**: [Solution]

## Important Concerns

### I1. [Title]
**File**: `path/to/file.ext:line`

[Issue description]

**Suggested change**: [Recommended approach]

**Trade-offs**: [Only if validator returned caveats]

## Minor Suggestions

### M1. [Title]
**File**: `path/to/file.ext:line`

[Current situation]

**Suggested change**: [Suggested improvement]

## Scope Concerns

### S1. [Title]
**File**: `path/to/file.ext:line`

[Description of scope concern]

**Suggested change**: [What to confirm or change]

## Addressed External Findings
5 issues from external reviewers were marked resolved or addressed by the PR author.

### R1. [CodeRabbit] Sort inversion in comparator
**File**: `OrderService.php:176`
**Status**: Resolved - author replied "Fixed in latest commit"

### R2. [CodeRabbit] Argument swap in getOrdersNeedingFulfillment
**File**: `Order.php:1230`
**Status**: Thread marked resolved

## Checklist
- Database: N/A
- Test Coverage: WARN
- Performance: PASS
- Documentation: WARN
- Security: N/A
- Code Quality: WARN
- Scope: PASS

## Assessment
[Concluding paragraph about overall state]

## Validation
- Pass 1 items: 11
- Bugfinder items: 0
- Kept: 8
- Clarify: 2
- Removed: 1
```

## Parsing Rules

- **Frontmatter**: YAML between `---` markers. `risk` is integer 1-10. `recommendation` is one of: `approve`, `request-changes`, `needs-discussion`. `head_ref` is the PR head branch name (used to construct GitHub file links). `pr_url` is the full PR URL.
- **Sections**: `## Heading` marks a section. Section names are exact: Intent, Changes, Risk, Results, Positives, Critical Issues, Important Concerns, Minor Suggestions, Scope Concerns, Addressed External Findings, Checklist, Assessment, Validation.
- **Findings**: `### ID. Title` starts a finding. ID format: `C1`, `I1`, `M1`, `S1`, `BF-1`, or `R1`.
- **File line**: `**File**: \`path:line\`` immediately after the finding heading.
- **Bugfinder line**: `**Bugfinder**: Confirmed (BF-N)` — optional, present only when the local bugfinder independently flagged the same issue. Appears after `**File**:` and before the body paragraph.
- **Status line**: `**Status**:` for R-prefixed (addressed external) findings - how the issue was resolved.
- **Body**: Everything between the File/Bugfinder lines and the next bold label is the problem/issue description.
- **Action labels**: `**Suggested change**:` is the standard label for all severity levels. `**Trade-offs**:` may follow if the validator returned caveats. Each label is on its own line followed by the content. Legacy labels (`**Fix**:`, `**Suggestion**:`, `**Alternative**:`, `**Action**:`) are also accepted by the converter for backwards compatibility.
- **Positives**: Simple bullet list with backticked file references.
- **Checklist**: `- Label: STATUS` where STATUS is PASS, WARN, FAIL, or N/A.
- **Validation**: `- Label: N` key-value pairs.
- **Empty sections**: If a severity has no findings, omit the section entirely. Addressed External Findings is also optional -- omit if there were no external review threads.
- **Addressed External Findings**: First line after heading is a summary sentence with the count. Entries use `R1`, `R2`, etc. IDs. Each has a `**File**:` and `**Status**:` line. No body paragraph needed.
- **Disclaimer**: Always include as the last line of the file, after all sections: `Warning: This report was generated by AI, and AI can make mistakes. Make sure to verify these findings before acting on them.`
